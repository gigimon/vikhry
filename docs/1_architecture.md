# Основные компоненты системы

1. Оркестратор - центральный компонент, который общается с Redis и управляет всеми worker'ами, а также предоставляет API для CLI и UI. Он отвечает за распределение задач между worker'ами, мониторинг их состояния и обработку результатов. Работает на основе uvloop и robyn (для web части).
2. Worker - отдельный процесс, который выполняет задачи, полученные от оркестратора. Именно он должен запускать тестовые сценарии, собирать метрики и отдавать их в Redis
3. Redis - используется для обмена данными между оркестратором и worker'ами.

# Структуры данных в Redis и схема взаимодействия

1. Пользователи - users
   - users - список со всеми id пользователей, которые были переданы воркерам
   - user:{user_id} - хэшмап. Для начала в нем хранить статус: {"status": "pending" | "running"} - pending когда передали воркеру, но он еще не начал его исполнять, статус ставится воркером
2. Ресурсы
   - resources - хэшмап, где ключ это название ресурса, а значение - количество
   - resource:{resource_name}:{id} - json со всеми данными
   - resource:{resource_name}:{id}:worker - id воркера, который взял этот ресурс (заполняет воркер)
3. Worker'ы
   - workers - set со всеми id воркеров, которые были запущены
   - worker:{worker_id} - канал (pub/sub), в который оркестратор будет отправлять команды, такие как: start/stop, get user <id>
   - worker:{worker_id}:users - список id пользователей, которые были переданы этому воркеру (заполняет воркер)
   - worker:{worker_id}:resources - список ресурсов, которые были взяты этим воркером (заполняет воркер)
   - worker:{worker_id}:status - healthcheck статус воркера (заполняет воркер), hashmap с полями: {"status": "healthy" | "unhealthy", "last_heartbeat": timestamp}
4. Метрики
    - metrics - set со всеми названиями метрик (уникальный id метрики: путь, метод)
    - metric:{metric_id} - stream, в который воркеры будут писать данные. Каждая запись в стриме - это hashmap с данными метрики

# Схема работы

## Инициализация компонентов
1. Оркестратор запускается и устанавливает соединение с Redis. Создает необходимые структуры данных, такие как: users, resources, workers, metrics. Раз в какое-то время, проверяет workers и обновляет у себя список активных воркеров
2. Запускается воркер, записывает свой id в workers и подписывается на канал worker:{worker_id}. В этот момент, оркестратор тоже подписывается на этот канал и создает все связаные списки
3. Воркеры в фоне начинают раз в секунду отправлять heartbeat в worker:{worker_id}:status, обновляя статус на healthy и timestamp.

## Запуск теста
1. Оркестратор получает команду от CLI/UI запустить тест на N пользователей. Начинает создавать ресурсы и записывать их в resources. После окончания создания ресурсов, записывает в users все id юзеров. Затем проверяет живые воркеры и распределеяет по ним юзеров, отправляя в worker:{worker_id} команду: add_user <id>
2. После того, как юзера распределились, отправляет всем воркерам команду: start_test
3. Воркеры создают нужное количество VU, отправляя метрики в Redis.

## Изменение количества юзеров
1. Если надо добавить пользователей, то оркестратор добавляет новых юзерво и отправляет команду воркерам, если удалить, то remove_user <id>, при получении подтверждения от воркера - удаляем из списка

## Остановка теста
1. Оркестратор отправляет всем воркерам команду: stop_test. Воркеры останавливают выполнение теста, удаляют всех юзеров и ресурсы, отправляя события в worker:{worker_id} канал, что они удалены и убирает из своих списков. Оркестратор удаляет всех юзеров и ресурсы из Redis.
