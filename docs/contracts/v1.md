# vikhry v1 contract

## Scope and constraints

1. Один сквозной run без `run_id`.
2. Одновременно может быть только один активный тест.
3. `lifecycle lock` между несколькими orchestrator-инстансами не используется.
4. Управляющие команды worker'ам отправляются через Redis Pub/Sub.
5. Broadcast-команды не используются: отправка только в персональный канал worker.
6. Подтверждения команд (`ack`/`nack`) не используются.
7. Масштабирование пользователей выполняется только через `add_user`/`remove_user`.
8. Распределение новых пользователей между worker'ами — `round-robin`.
9. Конфигурационный файл не используется: все runtime-параметры передаются через CLI.
10. Ресурсы не очищаются автоматически на `stop_test`, дубли ресурсов в v1 допустимы.

## Test state machine

`IDLE -> PREPARING -> RUNNING -> STOPPING -> IDLE`

Правила:
1. `start_test` разрешен только из `IDLE`.
2. `change_users` разрешен только в `RUNNING`.
3. `stop_test` разрешен в `PREPARING` и `RUNNING`.

## Redis keyspace

| Key | Type | Purpose |
|---|---|---|
| `test:state` | string | `IDLE | PREPARING | RUNNING | STOPPING` |
| `test:epoch` | integer | Увеличивается на каждый `start_test` |
| `workers` | set | Зарегистрированные `worker_id` |
| `worker:{worker_id}:commands` | pub/sub channel | Команды конкретному worker |
| `worker:{worker_id}:users` | set | Пользователи, назначенные worker'у |
| `worker:{worker_id}:status` | hash | Healthcheck: `status`, `last_heartbeat` |
| `users` | set | Все активные `user_id` теста |
| `user:{user_id}` | hash | `status`, `worker_id`, `updated_at` |
| `resources` | hash | Счетчик ресурсов по имени |
| `resource:{resource_name}:{id}` | string/json | Данные ресурса |
| `metrics` | set | Список `metric_id` |
| `metric:{metric_id}` | stream | События метрик |

## Metric event payload (v1)

Обязательные поля события метрики:
1. `name` - уникальное имя метрики (например, HTTP path `/page1`)
2. `step` - имя step, в контексте которого была собрана метрика
3. `status` - успешность (`true`/`false`)
4. `time` - длительность в миллисекундах

Дополнительно разрешены любые пользовательские поля (например `method`, `status_code`, `kind`).

## Worker healthcheck contract

Ключ: `worker:{worker_id}:status` (hash)

Поля:
1. `status`: `healthy | unhealthy`
2. `last_heartbeat`: unix timestamp
3. `cpu_percent`: загрузка процесса worker (проценты)
4. `rss_bytes`: RSS процесса worker (bytes)
5. `total_ram_bytes`: общий объем системной RAM (bytes)

## Command envelope (JSON)

```json
{
  "type": "add_user",
  "command_id": "uuid",
  "epoch": 1,
  "sent_at": 1761571200,
  "payload": {
    "user_id": 10
  }
}
```

Поля:
1. `type`: `start_test | stop_test | add_user | remove_user`
2. `command_id`: уникальный идентификатор команды
3. `epoch`: текущая эпоха теста (`test:epoch`)
4. `sent_at`: unix timestamp времени отправки
5. `payload`: параметры команды

## Command payloads

1. `start_test`: `{"target_users": <int>, "init_params": <object, optional>}`
2. `stop_test`: `{}`
3. `add_user`: `{"user_id": <int|string>}`
4. `remove_user`: `{"user_id": <int|string>}`

## Delivery model

1. Orchestrator публикует команды в `worker:{worker_id}:commands`.
2. Команды отправляются каждому worker отдельно, без broadcast.
3. `ack`/`nack` не ожидаются и не обрабатываются.

## User distribution (round-robin)

1. Берется список alive worker'ов в стабильном порядке.
2. Новый пользователь назначается следующему worker по кольцу.
3. Если alive worker'ов нет, операции запуска/добавления пользователей завершаются ошибкой.

## Lifecycle behavior

### `start_test`
1. Проверить `test:state == IDLE`, иначе вернуть ошибку.
2. Установить `test:state=PREPARING`.
3. Увеличить `test:epoch`.
4. Подготовить/дополнить ресурсы.
5. Отправить `start_test` каждому alive worker.
6. Раздать пользователей через `add_user` (round-robin).
7. Установить `test:state=RUNNING`.

### `change_users`
1. Допустим только при `test:state == RUNNING`.
2. На увеличение: серия `add_user`.
3. На уменьшение: серия `remove_user`.

### `stop_test`
1. Установить `test:state=STOPPING`.
2. Отправить `stop_test` каждому alive worker.
3. Очистить пользовательские ключи (`users`, `user:*`, `worker:*:users`).
4. Не очищать ресурсы автоматически.
5. Установить `test:state=IDLE`.

## Orchestrator HTTP API (v1)

Основные endpoint'ы:
- `GET /health`
- `GET /ready`
- `POST /start_test`
- `POST /change_users`
- `POST /stop_test`
- `POST /create_resource`
- `POST /ensure_resource`
- `GET /metrics`
- `GET /workers`
- `GET /resources`
- `GET /scenario/on_init_params`

### `GET /metrics`

Ключевые поля ответа:
1. `generated_at` - unix timestamp snapshot.
2. `lag` - backlog индикаторы (`detected`, `metrics_with_backlog`, `dropped_subscriber_messages`).
3. `metrics[]` - список метрик.

Поля `metrics[].aggregate`:
1. `window_s`
2. `requests`
3. `errors`
4. `error_rate`
5. `rps`
6. `latency_avg_ms`
7. `latency_median_ms`
8. `latency_p95_ms`
9. `latency_p99_ms`

`latency_*` поля возвращают `null`, если в окне нет latency sample.

### `GET /workers`

Ответ:
```json
{
  "generated_at": 1761571200,
  "count": 2,
  "workers": [
    {
      "worker_id": "w1",
      "status": "healthy",
      "last_heartbeat": 1761571198,
      "heartbeat_age_s": 2,
      "users_count": 120
    },
    {
      "worker_id": "w2",
      "status": null,
      "last_heartbeat": null,
      "heartbeat_age_s": null,
      "users_count": 0
    }
  ]
}
```

Поля worker:
1. `worker_id` - id worker.
2. `status` - `healthy | unhealthy | null` (если status еще не опубликован).
3. `last_heartbeat` - unix timestamp или `null`.
4. `heartbeat_age_s` - возраст heartbeat в секундах или `null`.
5. `users_count` - количество назначенных users (`worker:{worker_id}:users`).

### `GET /resources`

Ответ:
```json
{
  "generated_at": 1761571200,
  "count": 2,
  "resources": [
    { "resource_name": "accounts", "count": 100 },
    { "resource_name": "users", "count": 1000 }
  ]
}
```

Поля:
1. `generated_at` - unix timestamp формирования snapshot.
2. `count` - количество resource names в ответе.
3. `resources[]` - список счетчиков из Redis hash `resources`.

## Worker command handling rules

1. Невалидный JSON или неизвестный `type` игнорируется с логированием.
2. Обработчики команд должны быть идемпотентными.
3. Доставка команд best-effort (ограничение Pub/Sub).
4. Команды обрабатываются последовательно (по одной, в рамках одного event loop worker).
5. Переключение `epoch` у worker происходит только на `start_test`.
6. `add_user/remove_user/stop_test` выполняются только при `command.epoch == current_epoch`.
