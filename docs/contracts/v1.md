# vikhry v1 Contract

## Scope and constraints

1. There is no `run_id`; v1 assumes a single linear run model.
2. Only one active test may exist at a time.
3. No orchestrator lifecycle lock is implemented across multiple orchestrator instances.
4. Worker control commands are sent through Redis Pub/Sub.
5. Broadcast command delivery is not used; each worker has a personal channel.
6. `ack` and `nack` are not part of the protocol.
7. User scaling is limited to `add_user` and `remove_user`.
8. New users are distributed with round-robin across alive workers.
9. No external config file is used; runtime parameters come from CLI flags.
10. Resources are not automatically cleaned up on `stop_test`.

## Test state machine

`IDLE -> PREPARING -> RUNNING -> STOPPING -> IDLE`

Rules:
1. `start_test` is valid only from `IDLE`.
2. `change_users` is valid only from `RUNNING`.
3. `stop_test` is valid from `PREPARING` and `RUNNING`.

## Redis keyspace

| Key | Type | Purpose |
|---|---|---|
| `test:state` | string | `IDLE | PREPARING | RUNNING | STOPPING` |
| `test:epoch` | integer | Incremented on each `start_test` |
| `workers` | set | Registered worker IDs |
| `worker:{worker_id}:commands` | pub/sub channel | Command channel for one worker |
| `worker:{worker_id}:users` | set | Users assigned to a worker |
| `worker:{worker_id}:active_users` | set | Users that passed `on_init/on_start` |
| `worker:{worker_id}:status` | hash | Worker health payload |
| `users` | set | All active test user IDs |
| `user:{user_id}` | hash | Per-user assignment and state |
| `resources` | hash | Resource counters by name |
| `resource:{resource_name}:{id}` | string/json | Serialized resource payload |
| `metrics` | set | Known metric IDs |
| `metric:{metric_id}` | stream | Raw metric events |

## Metric event payload

Required fields:
1. `name`
2. `step`
3. `status`
4. `time`
5. `source`
6. `stage`
7. `result_code`
8. `result_category`
9. `fatal`

Custom fields may also be attached, such as:
- `method`
- `status_code`
- `error_type`
- `error_message`

## Worker healthcheck payload

Key: `worker:{worker_id}:status`

Fields:
1. `status`
2. `last_heartbeat`
3. `cpu_percent`
4. `rss_bytes`
5. `total_ram_bytes`

## Command envelope

```json
{
  "type": "add_user",
  "command_id": "uuid",
  "epoch": 1,
  "sent_at": 1761571200,
  "payload": {
    "user_id": 10
  }
}
```

Fields:
1. `type`
2. `command_id`
3. `epoch`
4. `sent_at`
5. `payload`

Supported command types:
- `start_test`
- `stop_test`
- `add_user`
- `remove_user`

## Command payloads

1. `start_test`: `{"target_users": <int>, "init_params": <object, optional>}`
2. `stop_test`: `{}`
3. `add_user`: `{"user_id": <int|string>}`
4. `remove_user`: `{"user_id": <int|string>}`

## Delivery model

1. The orchestrator publishes commands to `worker:{worker_id}:commands`.
2. Commands are delivered per worker, not through broadcast.
3. `ack` and `nack` are not expected.

## User distribution

1. The orchestrator takes alive workers in stable order.
2. Each new user is assigned to the next worker in the round-robin ring.
3. If no alive workers exist, startup or scale-up fails.

## Lifecycle behavior

### `start_test`

1. Validate `test:state == IDLE`.
2. Set `test:state=PREPARING`.
3. Increment `test:epoch`.
4. Prepare resources.
5. Send `start_test` to alive workers.
6. Send `add_user` commands using round-robin allocation.
7. Set `test:state=RUNNING`.

### `change_users`

1. Allowed only in `RUNNING`.
2. Scale-up sends `add_user`.
3. Scale-down sends `remove_user`.

### `stop_test`

1. Set `test:state=STOPPING`.
2. Send `stop_test` to alive workers.
3. Clear user-related keys.
4. Leave resources intact.
5. Set `test:state=IDLE`.

## Orchestrator HTTP API

Core endpoints:
- `GET /health`
- `GET /ready`
- `POST /start_test`
- `POST /change_users`
- `POST /stop_test`
- `POST /create_resource`
- `POST /ensure_resource`
- `GET /metrics`
- `GET /workers`
- `GET /resources`
- `GET /scenario/on_init_params`

### `GET /metrics`

Important fields:
1. `generated_at`
2. `lag`
3. `metrics[]`

Aggregate fields:
1. `window_s`
2. `requests`
3. `errors`
4. `error_rate`
5. `rps`
6. `latency_avg_ms`
7. `latency_median_ms`
8. `latency_p95_ms`
9. `latency_p99_ms`
10. `result_code_counts`
11. `result_category_counts`
12. `fatal_count`
13. `top_result_codes`

Latency fields may be `null` when no latency samples exist in the window.

### `GET /workers`

Example:

```json

  "generated_at": 1761571200,

    }
  ]
}
```

Worker fields:
1. `worker_id`
2. `status`
3. `last_heartbeat`
4. `heartbeat_age_s`
5. `users_count`

### `GET /resources`

Example:

```json
{
  "generated_at": 1761571200,
  "count": 2,
  "resources": [
    { "resource_name": "accounts", "count": 100 },
    { "resource_name": "users", "count": 1000 }
  ]
}
```

Fields:
1. `generated_at`
2. `count`
3. `resources[]`

## Worker command handling rules

1. Invalid JSON or unknown command types are ignored with logging.
2. Command handlers must be idempotent.
3. Pub/Sub delivery is best-effort.
4. Commands are processed sequentially in one event loop.
5. Workers switch epoch only on `start_test`.
6. `add_user`, `remove_user`, and `stop_test` require `command.epoch == current_epoch`.
