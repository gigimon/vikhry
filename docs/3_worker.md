# Алгоритм работы worker'а

## Технологии

Используется асинхронная модель исполнения с uvloop.  
Взаимодействие с Redis через `redis.asyncio`.  
Команды принимаются как JSON (`orjson`).

## Функциональность

- Подписка на персональный канал команд `worker:{worker_id}:commands`.
- Исполнение тестового сценария (VU).
- Обновление healthcheck статуса в Redis.
- Отправка метрик в Redis stream.
- Локальное управление назначенными пользователями (`add_user`/`remove_user`).

## Healthcheck

Worker периодически обновляет ключ:
- `worker:{worker_id}:status` (hash)
- поля:
  - `status`: `healthy | unhealthy`
  - `last_heartbeat`: unix timestamp

## Команды от orchestrator

Формат входящей команды:

```json
{
  "type": "add_user",
  "command_id": "uuid",
  "epoch": 1,
  "sent_at": 1761571200,
  "payload": {
    "user_id": 10
  }
}
```

Поддерживаемые типы:
1. `start_test`
2. `stop_test`
3. `add_user`
4. `remove_user`

## Правила обработки команд (v1)

1. Неизвестный `type` или невалидный JSON игнорируется с логом.
2. Команды обрабатываются без отправки `ack/nack`.
3. Обработчики должны быть идемпотентными:
   - повторный `add_user` не должен дублировать VU;
   - повторный `remove_user` не должен падать;
   - повторный `start_test`/`stop_test` не должен ломать state worker'а.

## Алгоритм работы

1. При старте worker подключается к Redis, регистрирует себя в `workers` и подписывается на `worker:{worker_id}:commands`.
2. Поднимает heartbeat loop и регулярно обновляет `worker:{worker_id}:status`.
3. Загружает тестовый сценарий из файла и ждет команд.
4. На `add_user` добавляет пользователя в локальный набор и запускает/готовит VU.
5. На `remove_user` останавливает VU и удаляет пользователя из локального набора.
6. На `start_test` переводит локальное исполнение в рабочий режим.
7. На `stop_test` останавливает все VU и очищает локальное состояние пользователей.

## Примерная структура кода

1. Основной класс `Worker` для event loop, Redis и диспетчеризации команд.
2. Класс `VU` для исполнения тестового сценария.
3. HTTP-клиент на базе `pyreqwest` для шагов сценария.
4. Класс работы с метриками (отправка событий в Redis streams).
5. Класс/модуль работы с ресурсами и выдачи их VU.
